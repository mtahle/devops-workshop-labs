Three core Docker concepts

Image → Container → Registry. This is the only vocabulary you need to get started.

Image

The blueprint

Read-only. Built once, stored forever. Layers stacked on top of a base like python:3.11-slim.

Think: a class definition. Not running — just a template.

Container

The running instance

A live, isolated process. One image can spawn 100 containers — each isolated from the others.

Think: a class instance. Running in its own memory space.

Registry

The warehouse

Docker Hub, Amazon ECR, GitHub Packages. Pull base images from here. Push your built images here.

Think: PyPI for images. docker pull / push

Dockerfile

Anatomy of a Dockerfile

FROM python:3.11-slim          # start from an official base image

WORKDIR /app                   # set the working directory inside the container

COPY requirements.txt .        # copy deps first — layer cache trick!
RUN pip install -r requirements.txt

COPY . .                       # then copy the rest of your code

EXPOSE 8000                    # document which port the app listens on

CMD ["python", "app.py"]       # default command when container starts
Terminal

Four commands you need today

# 1. Build an image from the Dockerfile in the current directory
$ docker build -t my-app:v1 .

# 2. Run a container (detached, port mapped host:container)
$ docker run -d -p 8000:8000 my-app:v1

# 3. List running containers
$ docker ps

# 4. Push to Docker Hub (replace with your username)
$ docker push myusername/my-app:v1
